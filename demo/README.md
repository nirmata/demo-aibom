# Nirmata AI Security Demo — Runbook

Three-pillar live demo: AI BOM supply chain visibility, AI Controls proxy governance, and Kyverno eBPF runtime enforcement.

---

## What this demo shows

| Pillar | Customer question | What Nirmata does |
|--------|-----------------|-------------------|
| **1 · AI BOM** | "How do I know what AI is in my code?" | `nctl` scans source → discovers agents, tools, models, MCP; CI gate blocks unapproved additions |
| **2 · AI Controls Proxy** | "How do I ensure agents only call approved LLMs?" | Inline proxy intercepts every LLM API call and enforces a sanctioned model list |
| **3 · Kyverno eBPF Runtime** | "What if someone bypasses the proxy?" | eBPF hooks into kernel `connect()` syscalls — no userspace bypass possible |

---

## How the demo is split

| | Pillar 1 | Pillars 2 & 3 |
|---|---|---|
| **Where** | GitHub Actions | Local terminal |
| **Why** | `nctl agent aibom` requires Linux CGO build — macOS binary excludes scanner | `kubectl` and Kyverno work on macOS |
| **Script** | Actions → **AIBOM Demo — Pillar 1** → Run workflow | `bash demo/run-demo.sh` |

## Prerequisites

| Tool | Purpose | Install |
|------|---------|---------|
| `kind` | Local Kubernetes cluster | https://kind.sigs.k8s.io/docs/user/quick-start/ |
| `docker` | Build images, run local registry | https://docs.docker.com/get-docker/ |
| `kubectl` | Interact with the cluster | https://kubernetes.io/docs/tasks/tools/ |
| `helm` | Install Kyverno | https://helm.sh/docs/intro/install/ |
| `cosign` | Attest images | https://docs.sigstore.dev/cosign/system_config/installation/ |
| `nctl` *(Pillar 1 via CI)* | AIBOM scanning and diff gate | Contact nirmata.com — private preview |

> **macOS note:** The macOS `nctl` binary does not include the AIBOM scanner (requires
> CGO + `include_aibom_scanner` build tag). Pillar 1 always runs in GitHub Actions
> on a Linux runner where `nctl` is fully supported.

## Quick start

### Step 1 — Clone the reference repo

`setup-demo-cluster.sh` **must be run from the repo root** (not from `demo/`). It depends on `Dockerfile`, `src/research-agent.ts`, `aibom-baseline.json`, and `policies/`.

```bash
git clone https://github.com/nirmata/kyverno-aibom-reference
cd kyverno-aibom-reference
```

### Step 2 — Set up the local cluster (once per demo machine)

```bash
bash demo/setup-demo-cluster.sh
```

Takes **3–5 minutes**. Run this before the customer call. Creates:

- kind cluster `aibom-demo` with local OCI registry on `localhost:5001`
- Kyverno (via Helm) with `ImageValidatingPolicy` CRD
- cosign keypair in `demo/keys/`
- `research-agent` image built, pushed, and AIBOM-attested
- `ai-controls` and `agents` namespaces
- Pillar 3 policies pre-applied (`NetworkPolicy` + `ClusterPolicy`)

### Step 3 — Show Pillar 1 via GitHub Actions

Before or during the call, trigger from the repo's **Actions** tab:

1. Go to **Actions → AIBOM Demo — Pillar 1: AI BOM Supply Chain**
2. Click **Run workflow**
3. Choose a scenario:

| Scenario | What it shows |
|----------|--------------|
| `approved` | Scan → gate passes → SARIF → attest → publish to NCH |
| `bad-tool` | `filesystem` tool detected → diff gate **BLOCKS** before build runs |
| `bad-model` | `gpt-5` detected → diff gate **BLOCKS** before build runs |

Walk the customer through the job graph in the Actions UI.

### Step 4 — Run Pillars 2–3 in the terminal

```bash
bash demo/run-demo.sh
```

The script pauses between sections. Press **Enter** to advance.

## Script reference

### `setup-demo-cluster.sh`

One-time bootstrap. Run before the demo, not during it.

| Step | Detail |
|------|--------|
| Prerequisite check | Verifies `kind`, `docker`, `kubectl`, `helm`, `cosign`; checks nctl aibom support |
| Cleanup | Tears down any previous `aibom-demo` cluster and registry container |
| Local OCI registry | Starts `registry:2` on `localhost:5001`, connected to kind Docker network |
| kind cluster | Creates `aibom-demo` with containerd configured to trust the local HTTP registry |
| Kyverno | Installs via Helm; applies `ImageValidatingPolicy` CRD from GitHub |
| cosign keypair | Generates `demo/keys/cosign.{key,pub}` — unencrypted, no Sigstore infrastructure |
| Build + push | Builds `research-agent` image, pushes to local registry |
| AIBOM | Runs `nctl agent aibom generate` (Linux) or uses `aibom-baseline.json` (macOS) |
| Attestation | `cosign attest` — AIBOM cryptographically bound to image digest |
| Namespaces | Creates `ai-controls` (Pillar 2) and `agents` (Pillar 3) namespaces |
| Pillar 3 policies | Applies `demo/ebpf-policies/agent-egress.yaml` (NetworkPolicy + ClusterPolicy) |
| Pre-pull images | Loads `curlimages/curl:latest` and `research-agent` into kind nodes |
| Smoke test | Verifies NetworkPolicy and ClusterPolicy are active |

**What gets created locally:**
```
demo/keys/cosign.key              private key (do not share)
demo/keys/cosign.pub              public key
demo/policies-local/              Kyverno ImageValidatingPolicies (for reference)
demo/aibom-demo.json              AIBOM predicate
demo/manifests/
  pod-approved-local.yaml         approved pod with real digest pre-filled
  pod-no-attestation-local.yaml   unattested pod for reference
```

**Tear down:**
```bash
kind delete cluster --name aibom-demo
docker rm -f aibom-demo-registry
rm -rf demo/keys demo/policies-local demo/aibom-demo.json
```

## Troubleshooting

**`nctl agent aibom` not supported on macOS**
Expected. The AIBOM scanner requires a Linux CGO build. Pillar 1 runs in GitHub Actions (Linux runner).

**Kyverno ClusterPolicy not auto-labelling pods**
Check the policy status:
```bash
kubectl get clusterpolicy label-ai-agent-pods
kubectl describe clusterpolicy label-ai-agent-pods
```
Kyverno needs a moment after apply — wait 15s and retry.

**NetworkPolicy not showing after setup**
```bash
kubectl get networkpolicies
kubectl apply -f demo/ebpf-policies/agent-egress.yaml
```

**AIControlsPolicy CRD not found (Pillar 2)**
Expected on a cluster without the AI Controls component deployed. The demo script gracefully skips the `kubectl apply` and shows the policy file for the talking point.

**cosign attest fails**
Ensure the registry container is connected to the kind network:
```bash
docker network connect kind aibom-demo-registry
```

**Starting fresh**
```bash
kind delete cluster --name aibom-demo
docker rm -f aibom-demo-registry
rm -rf demo/keys demo/policies-local demo/aibom-demo.json
bash demo/setup-demo-cluster.sh
```
